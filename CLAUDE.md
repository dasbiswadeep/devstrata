## What this repo is
devstrata is a configuration and documentation layer that wires 11 AI development
tools into a coherent stack. You are helping maintain this meta-package.

## Key rules for working in this repo

### Never modify upstream tool source
This repo contains configs, scripts, and docs — not tool implementations.
If something needs changing in Graphify, Mem0, etc., direct the user to
open an upstream issue/PR.

### Check for drift before editing install.sh
Install commands change frequently upstream. Before editing scripts/install.sh,
verify the current install command in the upstream README. Run
`./scripts/update.sh` to see current drift status.

### Personal project files do NOT belong in this repo
Personal project folders and personal config live OUTSIDE devstrata. Do not
import personal daily-flow docs, personal security configs, project-specific
entity models, or any personal context into this repo. devstrata is a generic,
open-source stack. Personal config belongs in the consuming project's AGENTS.md
or a separate personal repo — never here.

### KNOWN_ISSUES.md is first-class
Any change that introduces a new limitation or known failure mode MUST
update KNOWN_ISSUES.md before the PR is complete.

### SOURCES.md is first-class
Any factual claim about an upstream tool (star count, benchmark, license,
install command) MUST have a row in docs/SOURCES.md with the source URL and
the date verified. If you can't verify it, don't claim it.

### Memory domain discipline
When editing MEMORY_DOMAINS.md or ARCHITECTURE.md, preserve the domain split:
- Mem0 = semantic long-term facts
- Graphify = codebase structure
- Hermes FTS5 = conversation history
- Obsidian = human-curated knowledge
Do NOT blur these boundaries.

### Single source of truth for .mcp.json
There is ONE canonical .mcp.json: `configs/.mcp.json.template` (all 6 servers).
install.sh generates the profile-specific copy by stripping helix+mem0 for
--lite via jq. Do NOT create per-profile .mcp.json files — they drift. If you
need a per-profile server difference, edit the jq filter in install.sh, not
a new file.

## Common tasks in this repo

### Update a config template
```
Edit configs/AGENTS.md.template or configs/.mcp.json.template
Update INSTRUCTIONS.md to match
Update SOURCES.md verified date if install commands changed
Run ./scripts/test.sh to verify nothing broke
```

### Add a new tool to the stack
```
1. Verify tool license compatibility (MIT/Apache preferred)
2. Add to README.md table
3. Add to ARCHITECTURE.md layer diagram
4. Add setup steps to INSTRUCTIONS.md
5. Add known limitations to KNOWN_ISSUES.md
6. If it affects memory: update MEMORY_DOMAINS.md
7. If it affects security: update SECURITY.md
```

### Fix an upstream compatibility break
```
1. Identify which tool changed and what broke
2. Update the relevant script/config
3. Add to KNOWN_ISSUES.md if partially broken
4. Update SOURCES.md verified date + the relevant row
5. Test with ./scripts/test.sh && ./scripts/doctor.sh
```

## Repository layout
```
docs/       ← documentation (incl. SOURCES.md, SUPERVISION.md)
scripts/    ← install.sh, doctor.sh, morning-startup.sh, end-of-day.sh, update.sh, test.sh,
              sync-memory.sh, recommend-profile.sh, version-check.sh, headroom-watchdog.sh,
              validate-mcp.sh, agent-isolate.sh, wsl2-check.sh (13 scripts total)
configs/    ← AGENTS.md.template, .mcp.json.template (canonical), .graphifyignore.template,
              headroom.env.template, .gitignore.template, docker-compose.yml,
              com.helixdb.dev.plist, helixdb.service, com.devstrata.headroom.proxy.plist,
              com.devstrata.headroom-watchdog.plist, headroom-proxy.service,
              headroom-watchdog.timer (12 config templates)
profiles/   ← lite/, full/, pro/  (each has PROFILE.md only; .mcp.json is generated by install.sh)
tests/      ← test_*.sh — structural + integration tests (run via scripts/test.sh)
```
Config templates live in `configs/`. PROFILE.md files live in `profiles/<name>/`.
The .mcp.json is GENERATED from the single template — never copy a per-profile copy.

## Testing changes
```bash
./scripts/test.sh          # run the full test suite (62 test files, ~614 assertions, <5s)
./scripts/doctor.sh        # runtime health check against live services
./scripts/install.sh --lite # full install test (has interactive prompt)
```

The test suite is static analysis — it verifies the meta-package is internally
consistent (files exist, configs valid, scripts parse, no personal data, claims
sourced, profiles complete). Runtime health (are services up?) is doctor.sh.
